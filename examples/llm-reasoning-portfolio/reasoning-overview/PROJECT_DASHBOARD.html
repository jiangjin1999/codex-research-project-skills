Usage Guidelines

1 · Big-project idea — the four portfolio areas

The whole portfolio is split into four top-level areas. Put each thing in its narrowest home.

reasoning-overview/
Shared entry: dashboard, requirements log, templates, and the published dashboard copy.
reasoning-project/
One folder per project (copied from _PROJECT_TEMPLATE). The registry mirrors the Project Board.
reasoning-data/
Global/raw data, inventory, access boundaries, governance. Project-derived data is not written back here.
reasoning-ai/
Cross-project AI context: handoffs and global prompts that must survive one conversation.

2 · Single-project iteration — the six template folders

Every project shares the same skeleton so its map and this dashboard stay consistent.

0-Project/
Concrete attempts only — each a dated subfolder with its own data/docs/outputs/reports/scripts + TASK_BOARD.md.
1-Docs/
Design, methods, comparisons. 1-Docs/README.md is the fixed index.
2-Data/
Data notes only in 2-Data/DATA.md. Derived data stays here, never written back to global data.
3-Paper_Survey/
Literature + evidence: claim, evidence, effect on this project's decisions.
4-Skills/
Project-local prompts/procedures in 4-Skills/PROJECT_SKILLS.md.
_ai/
AI memory: project_overview, project_board_spec, task_plan, findings, progress.

3 · Behavior norms

Everyday use

  • Update Markdown / _ai first, then the board, then this dashboard.
  • Status needs an evidence path before done / paused / blocked.
  • Keep next action singular and concrete.
  • No raw records, secrets, or private excerpts anywhere public.

Initialization

  • Confirm Git identity before creating or editing a project.
  • Copy _PROJECT_TEMPLATE → reasoning-project/<name>.
  • Fill _ai/project_overview.md, add a registry row, then sync this dashboard (Level 3).
  • Initialize the project map + public copy under public-dashboard/projects/<slug>/.
